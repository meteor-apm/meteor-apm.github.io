Reduce CPU and network usage

Each subscription will create one or many observers. Observers initially fetch data from the DB and send to the client. If you are having a lot of subscriptions, there will be a lot of observers and they will impact on the CPU and Network.

But, Meteor could reuse observers for identical queries. You can track the Observer Reuse from our dashboard directly.

Then you can sort, publication with Low Observer Reuse & apply fixes with our guides.

Better utilization of server memory

Meteor maintains a copy of all your data sent to each client in the memory. It will be used to detect exact changes which needs to be sent to the client. But if you've a lot client or sending a lot of data, your app will drain server Memory.

We'll show you a very unique metric called Update Ratio which shows you the percentage of data change compared with all the data sent to the client.

If the Update Ratio is low, you actually don't need to keep them in the server Memory. So you can pick publication which sends a lot of data and having low update ratio; then apply our guides on how to avoid caching data on the server.

Improve user experience

Normally, Iron Router unsubscribe all the current route's subscriptions, before moving into another routes. Then, for each route change users needs to wait until data comes back from the server. Additionally, this will increase add unnecessary server load as well.

You can track those publications by sorting Shortest Lifespan and use our guidelines on improving this behaviour.
